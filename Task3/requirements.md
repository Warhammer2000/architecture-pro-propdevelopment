# Требования к внешним интеграциям «Умный дом»

## 1. Обзор

PropDevelopment внедряет две новые бизнес-функции через партнёра:
- **Интеллектуальный домофон** — классический видеодомофон + биометрическое распознавание по лицу с автоматическим открытием.
- **Интеллектуальный шлагбаум** — управление + распознавание номеров автомобилей (ANPR) с автоматическим открытием.

Обе функции встраиваются в существующее мобильное приложение собственников и работают поверх платформы партнёра, которая управляет физическими устройствами на объектах.

Ключевые особенности, влияющие на требования:
- Биометрические данные (лицо) — специальная категория ПДн по 152-ФЗ.
- Данные о номерах авто — также ПДн (относятся к идентификации лица).
- Устройства физически управляют доступом на территорию и в подъезд — отказ/подмена команды → прямая угроза безопасности жильцов.
- Интеграция с партнёром повторяет архитектурный паттерн, который уже создал проблемы с API управляющих компаний (кросс-доступ между УК). Это нужно сразу исключить.

## 2. Требования к безопасности

### 2.1. Классификация данных
| Данные | Категория | Основание |
|---|---|---|
| Биометрические шаблоны (embeddings лица) | Секретные | ст. 11 152-ФЗ — биометрические ПДн |
| Оригинальные изображения с камеры | Секретные | ст. 11 152-ФЗ |
| Номера автомобилей | Конфиденциальные | ст. 3 152-ФЗ, идентифицируют владельца |
| Журнал событий «кто вошёл/выехал» | Конфиденциальные | Формируют профиль перемещений собственника |
| Права доступа (списки разрешённых лиц/авто) | Конфиденциальные | Раскрытие = карта уязвимостей объекта |

### 2.2. Обработка ПДн
- Согласие на обработку биометрии — отдельное, письменное/электронное с УКЭП/УНЭП, с возможностью отзыва.
- Сбор только с согласия собственника; для гостей — отдельный поток (разовое приглашение со временем жизни).
- Хранение биометрических шаблонов — в виде необратимых embeddings, без оригинальных изображений сверх срока распознавания.
- Срок хранения = срок действия права доступа + 30 дней (retention policy), затем автоматическое безопасное удаление.
- Геораспределение хранения — только на территории РФ (152-ФЗ, ст. 18).

### 2.3. Защита данных в покое и в канале
- Шифрование БД с биометрией: AES-256, ключи в KMS/HSM, ротация не реже 1 раза в год.
- Шифрование канала между PropDevelopment и платформой партнёра: **mTLS** (минимум TLS 1.3, взаимная аутентификация по сертификатам).
- Шифрование канала от мобильного приложения: TLS 1.3 + certificate pinning.
- Мобильное приложение: локальное хранение токенов и кэша — в Secure Enclave / Keystore, без сохранения биометрии на устройстве.
- Подпись и верификация webhook-ов от партнёра (HMAC-SHA256 или подпись JWT).

### 2.4. Изоляция и сегрегация
- **Каждый собственник видит и управляет только теми устройствами, которые относятся к его квартире/жилому комплексу.** Проверка — на уровне авторизации, без опоры на клиентскую сторону.
- **Партнёр «Умного дома» получает данные и команды только по тем объектам, которые он обслуживает.** Multitenancy-токены с обязательным claim `complex_id`.
- Сетевая изоляция: trafficking к платформе партнёра — отдельный egress с явным allow-list, Network Policy в Kubernetes, выделенный VLAN в ЦОД.
- Сегрегация от остальных сервисов: `smart-home-app` в отдельном namespace, с политикой «default-deny» и явными allow-правилами (паттерн из Task5).

### 2.5. Защита от злоупотреблений
- Rate limiting на команды открытия: не более N команд/мин с одной квартиры.
- Anomaly detection: оповещение при массовых отказах распознавания, при распознавании в нетипичное время, при росте числа неизвестных лиц на камере.
- Идемпотентность команд открытия (idempotency key в запросе → защита от replay).
- Защита от replay на уровне канала: nonce + timestamp, TTL ≤ 30 с.
- Защита мобильного клиента от компрометации: jailbreak/root-detection, SafetyNet/DeviceCheck.

### 2.6. Аудит и мониторинг
- Все события доступа (попытки, успехи, отказы) логируются с полями: `who`, `what`, `when`, `where`, `device_id`, `decision`, `reason`.
- Логи отправляются в централизованный SIEM (см. проверочный лист, раздел IV).
- Доступ к журналам — только роли `security-ops` и `audit`.
- Ретенция логов доступа — не менее 1 года (для расследования инцидентов).
- Отдельный журнал доступа к биометрии (кто/когда читал embeddings) — обязательное требование 152-ФЗ.

### 2.7. Соответствие и ответственность
- Договор с партнёром должен включать: поручение на обработку ПДн (ст. 6 ч. 3 152-ФЗ), обязанность уведомлять об инцидентах ≤ 24 ч, право на аудит партнёра.
- DPIA (оценка воздействия на конфиденциальность) — до запуска пилота.
- Уведомление РКН об обработке биометрии.
- SLA на инциденты ИБ: время реакции партнёра ≤ 1 час, полное устранение ≤ 24 ч.

## 3. Аутентификация и авторизация

### 3.1. Собственник (пользователь → PropDevelopment)
- Протокол: **OAuth 2.0 + OpenID Connect** (Authorization Code + PKCE для мобильного клиента).
- Поставщик: существующий IdP PropDevelopment (через службу каталогов домена «Финансы», расширенную на ЖКУ — см. Task2 п. I.14).
- Второй фактор: обязательный MFA при управлении устройствами (push-уведомление в приложении или TOTP).
- Токены: short-lived access token (≤ 15 мин) + refresh token (≤ 30 дней, с возможностью немедленного отзыва).
- Claims в ID-токене: `sub`, `apartments[]` (список квартир с правами), `complex_ids[]`, `roles[]`.

### 3.2. PropDevelopment → платформа партнёра (machine-to-machine)
- Протокол: **OAuth 2.0 Client Credentials Flow** поверх mTLS.
- Токены: JWT, TTL ≤ 5 мин, без refresh (новый получается автоматически).
- Scopes:
  - `smart-home:door:open`
  - `smart-home:door:command`
  - `smart-home:barrier:open`
  - `smart-home:biometry:enroll`
  - `smart-home:biometry:revoke`
  - `smart-home:audit:read`
- Ключи клиента — в KMS, ротация не реже 90 дней.
- Каждый запрос к партнёру содержит claim `complex_id` — партнёр обязан валидировать принадлежность устройства этому ЖК.

### 3.3. Платформа партнёра → PropDevelopment (webhook events)
- Канал: mTLS от партнёра.
- Подпись webhook: HMAC-SHA256 либо signed JWT в заголовке.
- Замена endpoint-а с ротацией секрета при каждом пересогласовании.
- Валидация timestamp для защиты от replay.

### 3.4. Модель прав внутри PropDevelopment
- ABAC (атрибутно-ориентированная модель) — каждое обращение авторизуется по атрибутам: `user.apartments`, `device.apartment`, `device.complex_id`, `action`.
- Явный default-deny.
- Policy-as-code (например, OPA) для проверки решений — также используется в Task7.

## 4. Взаимодействие систем

### 4.1. Поток «собственник управляет домофоном/шлагбаумом»
1. Мобильное приложение → API Gateway партнёров (REST, HTTPS/TLS 1.3, OAuth2 access token).
2. API Gateway валидирует токен, применяет rate limit, пробрасывает в `smart-home-app`.
3. `smart-home-app` проверяет ABAC-политику, достаёт из `smart-home-db` права и `device_id`, вызывает API партнёра.
4. API партнёра выполняет команду на физическом устройстве и синхронно возвращает результат.
5. `smart-home-app` публикует событие `door.opened` в event-bus (Kafka).
6. Подписчики (audit-log, tenant-core-app) реагируют на событие.

### 4.2. Поток «гость/жилец распознаётся устройством»
1. Домофон/шлагбаум делает кадр → отправляет в платформу партнёра.
2. Платформа распознаёт и сверяет со списком разрешённых.
3. При успехе — открывает, публикует webhook в PropDevelopment.
4. PropDevelopment валидирует подпись, проксирует событие в event-bus.
5. `smart-home-app` сохраняет факт доступа в `smart-home-db`, отправляет push-уведомление собственнику.

### 4.3. Поток «собственник добавляет биометрию жильца»
1. Собственник в приложении даёт согласие + загружает фото.
2. Приложение отправляет изображение → API Gateway → `smart-home-app`.
3. `smart-home-app` **не сохраняет оригинальное изображение**, передаёт в платформу партнёра по mTLS.
4. Партнёр формирует embedding, возвращает `biometry_id`. Оригинал изображения на стороне партнёра хранится только на время формирования embedding.
5. `smart-home-app` сохраняет `biometry_id` + право доступа в `smart-home-db`, связывает с квартирой.

### 4.4. Обработка отказов и деградация
- При недоступности платформы партнёра: мобильное приложение показывает деградированный UX («удалённое открытие временно недоступно»), физическое открытие по ключу продолжает работать.
- Circuit breaker на уровне API Gateway партнёров: при росте ошибок > 50% за 1 мин — временная отсечка с exponential backoff.
- Dead-letter queue в event-bus для событий, которые не удалось обработать.
- Идемпотентность команд — повтор безопасен.

### 4.5. Сетевые требования
- Все связи с партнёром — только через единый API Gateway партнёрских интеграций (решает проблему из описания, когда разные команды интегрируются с партнёрами по-своему без единой политики).
- Allow-list IP-адресов партнёра на firewall.
- Отдельный egress-шлюз в Kubernetes, изолированный NetworkPolicy.
- DNS-резолюция партнёра — только через корпоративный DNS с логированием.

## 5. Эксплуатационные требования

- **SLA доступности** партнёра: 99.9%, измеряется от PropDevelopment.
- **RTO** для `smart-home-app`: ≤ 15 минут; **RPO**: ≤ 5 минут.
- **Мониторинг**: метрики задержки команд, доля отказов распознавания, количество неуспешных попыток авторизации, алерты в дежурную ИБ-команду.
- **Регулярные проверки**: pentest интеграции раз в год, проверка ротации ключей ежеквартально, ревью ABAC-политик ежеквартально.
- **Процесс изменений контракта API партнёра**: версионирование, breaking changes — с уведомлением не менее чем за 90 дней.
