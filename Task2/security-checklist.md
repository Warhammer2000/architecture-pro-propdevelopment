# Проверочный лист безопасности бизнес-систем PropDevelopment

## 1. Идентификация систем

По диаграмме C4 и описанию компании выделены следующие бизнес-системы, подлежащие аудиту.

### Домен «Продажи» (клиентские сервисы)
| Система | Назначение | Характер работы с данными | Ключевые риски ИБ |
|---|---|---|---|
| Витрина продаж (монолит) | Подбор и бронирование недвижимости | Публичные данные об объектах + ПДн клиентов (форма бронирования) | Точка регистрации ПДн, внешний периметр |
| client-tour-app + БД | Онлайн-тур | Медиа, идентификаторы клиентов | Внешний доступ, сессии |
| client-mart-app + БД | Онлайн-сделка | ПДн (паспорт, СНИЛС), интеграция с госорганами | Повышенная категория ПДн, юридическая значимость |
| client-crm-app + БД | CRM клиентов | Агрегированные ПДн клиентов | Крупнейший агрегат ПДн → главная цель утечки |
| client-mart-estate-app + БД | Данные о недвижимости | Каталог объектов | Искажение → репутационный ущерб |

### Домен «ЖКУ» (сервисы собственников)
| Система | Назначение | Характер работы с данными | Ключевые риски ИБ |
|---|---|---|---|
| Витрина ЖКУ | Вход в сервисы собственников | ПДн собственников, баланс ЖКУ | Внешний периметр |
| tenant-core-app + БД | Предоставление услуг, Умный дом, ремонты | Финансовые данные, команды управления автоматикой и замками квартир | Критический — прямое влияние на физическую безопасность |
| CRM собственников + БД | Управление данными собственников | ПДн собственников | Инцидент с отображением чужих ФИО в ЛК (активная проблема) |
| API партнёров (УК) | Интеграция с управляющими компаниями | ПДн, платёжные данные | Нет единых политик ИБ, кросс-доступ между УК — активная проблема |

### Домен «Финансы»
| Система | Назначение | Характер работы с данными | Ключевые риски ИБ |
|---|---|---|---|
| accountant-service-1 + БД | Финансовый учёт | Финансовая отчётность, платежи | Коммерческая тайна, требования ФНС |
| Служба каталогов | Управление учётными записями | Учётные данные сотрудников | Компрометация → доступ к любой системе компании |

### Домен «Дата»
| Система | Назначение | Характер работы с данными | Ключевые риски ИБ |
|---|---|---|---|
| Хранилище (DWH) | Консолидация данных | Агрегированные ПДн, финансы | Центральная точка утечки |
| BI | Отчётность | Внутренние данные | Качество данных блокирует ML/AI |
| Отчётность | Регламентные отчёты | Финансы, операционные метрики | Целостность |

### Инфраструктура
- Собственный ЦОД + приватное облако + арендованные мощности у провайдера (гибрид).
- Межсетевой экран на внешнем периметре для клиентов и собственников.
- Резервное копирование всех БД.

## 2. Обоснование выбора разделов

В проверочный лист включены **все пять разделов** шаблона. Это обусловлено тем, что PropDevelopment проводит **первый комплексный аудит** и заявленные бизнес-цели (соответствие 152-ФЗ, предотвращение утечек, развитие BI/ML/AI, рост удовлетворённости клиентов) затрагивают каждый из разделов. Ниже — приоритеты и причины включения каждого раздела.

| Раздел | Приоритет | Почему включён |
|---|---|---|
| **I. Управление доступом** | **Высокий** | Прямо связан с активным инцидентом (клиенты видят чужие ФИО в ЛК) и проблемой кросс-доступа партнёров-УК. Нарушен принцип минимальных привилегий. |
| **II. Безопасность данных** | **Критический** | Ключевая проблема компании: нарушен контроль данных, нет классификации, нет Data Governance. Блокирует BI/ML/AI-инициативу и соответствие 152-ФЗ. |
| **III. Защита инфраструктуры** | **Высокий** | Гибридная инфраструктура (свой ЦОД + облако) увеличивает поверхность атаки. Требуется подтверждение сегментации, IDS/IPS и сканирования облака. |
| **IV. Управление инцидентами** | **Высокий** | На фоне недавней утечки у конкурента руководство ожидает готовности к инцидентам. Один ИБ-специалист на всю компанию → процесс должен компенсировать нехватку ресурсов. |
| **V. Специфические аспекты** | **Средний** | Мобильное приложение собственников, собственный ЦОД и планируемая интеграция «Умного дома» требуют отдельной проверки (MDM применим частично — к устройствам сотрудников). |

Вопросы, по которым данных из описания кейса недостаточно, отмечены как «требует проверки» — это рабочие пункты для интервью с командами на этапе самого аудита.

## 4. Следующие шаги
